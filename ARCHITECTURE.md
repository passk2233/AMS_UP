## The GetX Pattern

Every feature module follows the same three-folder structure, generated by `get_cli` and kept consistent throughout the project:

```
modules/<feature>/
├── bindings/
│   └── <feature>_binding.dart      # Dependency injection: Get.lazyPut / Get.put
├── controllers/
│   └── <feature>_controller.dart   # State + business logic (GetxController)
└── views/
    └── <feature>_view.dart         # Stateless widget using GetView<T>
```

**Why GetX?** It bundles state management, navigation, dependency injection, and reactive primitives (`Rx`, `Obx`, `RxList`) into one cohesive API, which keeps boilerplate low and the learning curve shallow for newcomers.

### Reactive state primitives used

| Primitive | Use in this codebase |
|---|---|
| `Rx<T>` | Single observable value (e.g. `Rx<UserModel?> currentUser`) |
| `RxBool` / `RxInt` / `RxString` | Loading flags, counters, search queries |
| `RxList<T>` | Lists rendered with `Obx(() => ListView(...))` |
| `Obx(() => ...)` | Auto-rebuilding widget |
| `GetView<C>` | Stateless view bound to a controller via `Get.find()` |
| `Get.put` / `Get.lazyPut` | Controller registration in a `Bindings` class |
| `Get.toNamed` / `Get.offAllNamed` | Navigation with named routes |

## Role-Based Shells

After login, `AuthController` inspects the user's roles and pushes one of three shells:

| Role | Route | Shell |
|---|---|---|
| Administrator / Admin | `/admin-home` | `AdminShellView` + `AdminBottomNavBar` |
| Teacher | `/teacher-home` | `TeacherShellView` + `TeacherBottomNavBar` |
| Student | `/home-student` | `HomeStudentView` (uses `HomeStudentController.buildBottomNavigation()`) |

Each shell uses an `IndexedStack` (admin/teacher) or page swap (student) to keep tab state alive across navigation.

### Bottom nav controllers

Each shell has its own bottom-nav controller registered as **permanent** so it survives across hot reload and tab switches:

- `BottomNavController` (admin)
- `TeacherBottomNavController`
- `HomeStudentController` (does double duty: page switching + nav UI)

Tabs invoke `refreshData()` on the relevant page controller when activated, ensuring data is fresh without manual pull-to-refresh.

## Data Layer

Models live under `lib/app/modules/data/models/` and are re-exported through a barrel file:

```dart
// data_exporter.dart
export 'models/user_model.dart';
export 'models/room_model.dart';
// ... 30+ models total
```

Each model:

- Mirrors a database table from [`DATABASE_SCHEMA.md`](./DATABASE_SCHEMA.md).
- Implements `fromJson(Map<String, dynamic>)` and `toJson()`.
- Tolerates missing or null fields with `?? 0` / `?? ''` defaults to survive backend variance.
- Nests related models when the backend preloads them (e.g. `StudyPlanModel.teacher`, `RoomBookingModel.user`).

Usage in any module:

```dart
import '../../data/data_exporter.dart';

final user = UserModel.fromJson(response.data);
```

## Networking

All HTTP traffic goes through **Dio**, configured per controller with these defaults:

```dart
Dio(BaseOptions(
  baseUrl: dotenv.env['API_URL'] ?? '',
  connectTimeout: const Duration(seconds: 10),
  receiveTimeout: const Duration(seconds: 10),
  headers: {
    'Content-Type': 'application/json',
    'ngrok-skip-browser-warning': 'true',
  },
));
```

The `ngrok-skip-browser-warning` header lets development builds reach an ngrok-tunneled backend without the HTML interstitial.

### Auth header

The bearer token is loaded from `shared_preferences` before each batch of requests:

```dart
final prefs = await SharedPreferences.getInstance();
final token = prefs.getString('token') ?? '';
_dio.options.headers['Authorization'] = 'Bearer $token';
```

### Error handling

Every controller catches `DioException` and routes to the shared `AppDialogs.showError(...)` helper, which builds a human-readable detail string via `AppDialogs.buildDioErrorDetail(e)`. On 401 the user is force-logged-out to `/auth`.

See [API.md](./API.md) for the full list of endpoints consumed.

## Authentication Flow

```
┌──────────────────┐
│   AuthView       │  user enters username + password
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ AuthController   │  fetch FCM token (best-effort)
│   .login()       │  POST /auth/login { username, password, device_token, platform }
└────────┬─────────┘
         │
         ▼
   Response 200
   { token, user: { ..., roles: [...] } }
         │
         ▼
┌──────────────────────────────────────────┐
│  SharedPreferences                       │
│   token = <JWT>                          │
│   roles = ['Administrator' | ...]        │
│   saved_username (if Remember Me ticked) │
└────────┬─────────────────────────────────┘
         │
         ▼
   Routing decision based on roles
   ├── Administrator/Admin → /admin-home
   ├── Teacher             → /teacher-home
   └── Student             → /home-student
```

The token is **never** refreshed automatically — a 401 will boot the user back to the login screen. Refresh tokens are on the roadmap.

## Push Notifications (FCM)

Full cross-stack design for OS-visible foreground/background/terminated
delivery lives in [`../docs/fcm-notification-design.md`](../docs/fcm-notification-design.md).
That document is the implementation target for the system-notification-panel
requirement.

`lib/app/services/fcm_service.dart` is initialised in `main()` after Firebase:

```dart
await Firebase.initializeApp(options: DefaultFirebaseOptions.currentPlatform);
FirebaseMessaging.onBackgroundMessage(_firebaseMessagingBackgroundHandler);
await FCMService.init();
```

`FCMService.init()`:

1. Requests notification permission (iOS + Android 13+).
2. Subscribes to `FirebaseMessaging.onMessage` and shows a `Get.snackbar` toast.
3. Subscribes to `FirebaseMessaging.onMessageOpenedApp` for taps from background.
4. Reads `getInitialMessage()` for taps from terminated state.

Device tokens are sent up with login (`device_token`, `platform`) and stored in the `user_devices` table on the backend.

## Theming & UI Conventions

`lib/app/widgets/app_colors.dart` centralises the palette:

```dart
AppColors.primary         // 0xff40b4cd  — main teal
AppColors.laoBlue         // 0xFF4C4DDC  — accent indigo
AppColors.borderApproved  // 0xFF10B981  — green
AppColors.borderPending   // 0xFFF59E0B  — amber
AppColors.rejectRed       // 0xFFE53935  — red
AppColors.textPrimary     // 0xFF1A1A2E
AppColors.textSecondary   // 0xFF6B7280
```

Shared widgets in `lib/app/widgets/`:

- `AppDialogs` — `showSuccess`, `showWarning`, `showError`, `showConfirmation` (use these instead of `Get.snackbar` for important interactions).
- `BookingCard` — reusable booking row with approve/reject CTA.
- `ProfileCard` — admin dashboard summary card.
- `AdminAppBar` — gradient app bar with semester chip + pending-requests badge.
- `AdminBottomNavBar` — bottom-nav for admin.

All re-exported via `widgets/widget.dart`.

## Conventions Cheat Sheet

| Topic | Rule |
|---|---|
| Folder names | `snake_case` (legacy `Booking_student/` exempt) |
| File names | `snake_case.dart` |
| Classes | `PascalCase`, suffix matches role: `*View`, `*Controller`, `*Binding` |
| Reactive fields | `final ... = ....obs;` and exposed without setter |
| Async errors | Catch `DioException` separately, log via `debugPrint`, surface via `AppDialogs` |
| Routes | Defined as constants in `app_routes.dart`; use `Routes.X` not raw strings |
| API base URL | Always from `dotenv.env['API_URL']`, never hardcoded |
| Tokens | Read via `_loadToken()` in every controller before requests |
| Dates | Use `intl`'s `DateFormat`; render Lao weekday/month names where appropriate |
| Colors | Pull from `AppColors`, never inline `Color(0x...)` for primary palette |
